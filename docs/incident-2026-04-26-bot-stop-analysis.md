## 3. 근본 원인: STOP을 너무 쉽게 트리거하는 로직 + LLM 직관 미스매치

### 3-1. 코드 레벨의 백도어

```python
# main_agent.py _decide_action() (사고 당시)
if score >= 90:
    return "STOP"  # 극단적 스코어는 바로 STOP
```

문제: 1분봉 wick 한 번에 risk_score가 잠깐 90을 넘으면, 멀티 에이전트 합의도 거치지 않고 즉시 STOP. 그리드봇 본질에 반하는 백도어.

### 3-2. 멀티 에이전트가 "트레이더"로 사고

기존 4명 에이전트 (Technical/Sentiment/**Risk**/Macro)는 모두 **포지션 트레이더 페르소나**.

특히 Risk 에이전트는 "**자본 보전이 최우선**"으로 정의되어 있었고, 코디네이터에서 **STOP에 가중치 2배**를 받았다. 결과적으로 STOP 쪽 편향이 구조적으로 강화됨.

```
기존 합의 규칙 (사고 당시):
2. 리스크 관리자가 STOP을 권고하면 가중치 2배 부여
```

### 3-3. LLM이 그리드봇 메커니즘을 이해 못 함

LLM은 인터넷의 트레이딩 글로 학습되어 "추세 추종" 사고가 박혀있다. **그리드봇은 정확히 반대 — 변동성 = 친구, 경계 근처 = 일하는 자리**. LLM 시스템 프롬프트에는 이 메커니즘이 명시되어 있지 않았다.

LLM이 본 입력:
```
현재가: 2308
그리드: 2000-2400
가격 위치: 상단 77%
ATR: 평소보다 약간 높음
```

LLM 사고 (트레이더 본능):
> "가격이 상단 근처. 변동성 약간 있음. 이탈 위험 → 안전하게 STOP."

현실:
> 봇이 정확히 그 자리에서 매수/매도 반복하며 돈 벌어야 하는 상황.

### 3-4. 이벤트 트리거 자체가 횡보장에 최악

`_detect_events()`의 첫 번째 트리거:
```python
if position_pct >= 80:
    events.append("가격이 그리드 상단 도달")
```

중심봇 80% 라인 = 2,320. 4/24~4/26 가격이 2,300~2,330 사이라 거의 매 틱 80% 라인을 ping-ping. → LLM 호출 폭증 → 잘못된 판단 누적 확률↑.

### 3-5. STOP에 안전 가드 없음

`_check_restart_allowed()`는 WIDEN/SHIFT_UP/SHIFT_DOWN만 체크. STOP은 통과. LLM이 STOP 결정하면 무조건 실행됨.

## 4. 개선 사항 (이 사고로부터)

### 그리드봇 헌법 제정 (`multi_agent.py`)

모든 에이전트의 시스템 프롬프트 최상단에 **공통 헌법** 박음:

```
【그리드봇 운영 헌법 — 모든 판단의 전제】

1. 메커니즘:
   - 이 봇은 가격이 위아래로 출렁일 때 매수/매도를 반복해서 돈을 번다.
   - 변동성은 친구다. 횡보는 최적 환경이다.
   - 가격이 그리드 경계 근처에 있다는 것은 "곧 체결될 자리"이며 "위험"이 아니다.

2. 디폴트는 MAINTAIN:
   - 액션을 바꾸려면 명확한 증거가 필요하다. 모호하면 무조건 MAINTAIN.

3. STOP은 거의 없다:
   - STOP 후보는 시스템적/외부 위험에 한정 (1시간 ±20% 폭락, 거래소 장애 등).

4. 정지의 비용:
   - STOP은 자본 보전이 아니라 자본 효율 손실이다.
```

### 4가지 프롬프트 원칙 적용

| 원칙 | 적용 위치 | 효과 |
|------|----------|------|
| **① 메커니즘 명시** | `GRID_BOT_CONSTITUTION` → 모든 에이전트 system 최상단 | LLM이 그리드봇을 트레이더 관점이 아닌 운영자 관점으로 해석 |
| **② MAINTAIN 디폴트** | 코디네이터 합의 규칙 + 개별 에이전트 가이드라인 | 모호하면 봇을 그대로 둔다 (입증 책임 전환) |
| **③ STOP 재정의** | `_ask_agent` 프롬프트 + 코디네이터 STOP 보호 장치 | "잠시 멈춤"이 아니라 "전략 영구 폐기"로 재정의 |
| **④ 봇 운영 컨텍스트** | `_build_fee_context` 확장 | 누적 수익/일평균/일일 기회비용을 LLM에 숫자로 주입 |

### 코드 레벨 STOP 보호 장치 (`multi_agent._coordinate`)

LLM이 STOP을 답해도 다음 조건을 모두 만족 안 하면 **코드 레벨에서 강등**:

- 4명 만장일치 STOP
- 모든 STOP 의견 신뢰도 8/10 이상

미충족 시 차선 액션(STOP 제외 다수결, 동률이면 MAINTAIN)으로 자동 강등.

```python
def _fallback_non_stop(self, opinions):
    non_stop = [o for o in opinions if o.action != "STOP"]
    ...
    # 동률이면 MAINTAIN 우선 (그리드봇 헌법 ②)
    priority = {"MAINTAIN": 0, "WIDEN": 1, "SHIFT_UP": 2, "SHIFT_DOWN": 3}
```

### 코드 레벨 백도어 제거

`main_agent._decide_action`의 `if score >= 90: return "STOP"` 라인 제거.
score가 극단적이어도 멀티 에이전트 합의 + 코드 STOP 가드를 거치도록 강제.

### Risk 에이전트 가중치 2배 제거

코디네이터 합의 규칙에서 "리스크 관리자 STOP 가중치 2배" 삭제.
모든 에이전트가 동등한 1표.

### Risk 에이전트 정의 재작성

> 기존: "자본 보전이 최우선"
> 수정: "자본 효율이 최우선. 봇을 멈추는 것은 자본 효율 손실이다."

### 4명 → 2명 체제 전환 (구조적 재설계)

원인 분석 결과 4명 트레이더 페르소나(Technical/Sentiment/Risk/Macro) 합의 자체에 편향이 있었다:
- 4명이 "보수적 안전 답"을 찾으면 자연스럽게 STOP 쪽으로 수렴
- LLM 5호출(에이전트 4 + 코디네이터)에 비해 가성비 낮음

**새 구조 (의도된 비대칭 합의):**

| 에이전트 | 역할 |
|----------|------|
| **운영자 (Operator)** | 액션을 제안 — 그리드봇 운영자 관점 |
| **비판자 (Exit Critic)** | 액션 변경에 반대하는 변호인 — MAINTAIN 강하게 옹호 |
| 조율자 (Coordinator) | 합의 도출 — 변경에는 만장일치 필요 |

**작동 원리:**
- 변경하려면 두 명을 모두 설득해야 함 (만장일치)
- 유지(MAINTAIN)는 한 명만으로 충분
- 비판자가 "그냥 두면 돼"라고 답하면 변경 안 됨
- 그리드봇 헌법 ②(MAINTAIN 디폴트)를 구조 자체에 박음

**부수 효과:**
- LLM 호출 5회 → 3회 (코디네이터 포함) → 멀티 에이전트 비용 약 40% 절감
- 응답 시간 단축 (병렬 호출 4 → 2)
- STOP 보호 장치 강화: 2명 만장일치 + 신뢰도 9+ 필요 (4명 체제의 8+보다 보수적)

### 봇 운영 컨텍스트 (헌법 ④)

`_build_fee_context()`에 다음 정보 추가:

```
=== 봇 운영 현황 (정지의 기회비용 인식용) ===
누적 실현 손익: +656.49 USDT
운영 기간(추정): 21.0일
일평균 실현 손익: +31.26 USDT/일
⚡ 정지 시 일일 기회비용: ~31.26 USDT/일
   (봇이 돌면 하루에 평균 이만큼 번다는 뜻 — STOP은 이만큼을 매일 포기)
```

LLM에게 "STOP의 비용"을 숫자로 보여줌 → STOP을 함부로 권고 못 함.

## 5. 멀티봇 환경 대비 — 텔레그램 알림 핸들링

### 배경

2-bot ladder 운영 중인데, 알림에 봇 식별자가 없어 "어느 봇 이벤트인지" 즉시 구분 어려움.

### 적용된 변경

1. **`Notifier` 클래스에 `bot_label` 필드 추가**
   - 모든 send() 호출 시 메시지 첫 줄에 `[ETH 2000-2400]` 같은 prefix 자동 추가
   - 빈 라벨이면 prefix 생략 (단일봇 호환)

2. **`main_agent`에서 라벨 자동 생성**
   - `_compose_bot_label()`: `{base} {lower}-{upper}` 형식 (예: `ETH 2000-2400`)
   - 우선순위: `config.BOT_LABEL` (사용자 별칭) > 런타임 그리드 범위 > config 기본값

3. **WIDEN/SHIFT 후 라벨 자동 갱신**
   - `_refresh_bot_label()`을 모든 grid 변경 시점에 호출
   - 그리드 범위가 바뀌면 라벨도 즉시 따라가서 다음 알림부터 새 prefix 적용

4. **멀티 에이전트 합의도 봇 라벨 prefix**
   - `format_consensus_for_telegram(result, bot_label=...)` — 헤더에 봇 라벨 노출
   - LLM 컨텍스트(`_build_market_context`)에도 `[봇: <label>]` 박힘

### 멀티봇 운영 시 알림 사용법

여러 봇을 동시에 운영하려면 각 봇마다 **별개의 main_agent 프로세스**를 띄우고 (현재 시스템은 단일 controller 가정), 각자의 `.env`에서 `SYMBOL`/`GRID_LOWER`/`GRID_UPPER`/`BOT_LABEL`만 다르게 두면 됨.

같은 텔레그램 채널을 공유해도 prefix로 구분되어:

```

[ETH 2000-2400] 🤖 멀티 에이전트 합의 [ETH 2000-2400]
[ETH 2000-2400] ✅ 그리드 범위 복귀
```

향후 단일 프로세스에서 N봇을 직접 관리하려면 `MultiBotOrchestrator` 같은 상위 클래스가 필요. 이는 별도 마일스톤.

## 8. 교훈

1. **LLM은 도메인 지식이 없으면 도메인 직관도 없다.**
   "그리드봇은 변동성으로 돈 번다"는 사실은 인간에게도 처음 들으면 반직관적인데, 이걸 시스템 프롬프트에 박아주지 않으면 LLM이 알 길이 없다.

2. **"보수적이라는 이름의 위험".**
   "확신 없으면 STOP"은 보수적인 듯 보이지만, 그리드봇에서는 가장 비싼 결정이다. 매일 일정 수익을 영구 포기하는 거니까.

3. **OKX `cancelType: 1`은 만능 단서가 아니다.**
   "manual stop"이라고 표시되지만, 우리 자신의 코드가 API로 호출했을 때도 이렇게 찍힌다. UI의 한 단어에 속아 디버깅 방향 잘못 잡을 수 있음.

4. **멀티봇 알림은 처음부터 라벨 prefix.**
   봇이 하나일 때는 굳이 필요 없지만, 한 번 운영해보면 어느 봇 알림인지 구분이 안 돼서 결국 추가하게 된다. 디폴트로 깔아두는 게 맞다.
